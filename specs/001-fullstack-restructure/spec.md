## User Scenarios & Testing *(mandatory)*

### User Story 1 - 后端数据持久化与REST API (Priority: P1)

用户登录系统后，所有数据操作（需求、任务、项目、版本、迭代的增删改查）通过标准 REST API 与后端交互，数据持久化到 MySQL 数据库，不再依赖前端内存 mock 数据。刷新页面后数据不丢失，多用户可共享同一数据源。

**Why this priority**: 这是全栈重构的基础。没有后端和 API，前端无法获取真实数据，所有其他功能都依赖于此。P1 完成后系统即可从"演示原型"升级为"可用系统"。

**Independent Test**: 启动后端服务后，通过 API 工具（如 curl 或 Postman）直接调用各接口，验证数据能正确创建、查询、更新和删除，数据库中有对应记录。

**Acceptance Scenarios**:

1. **Given** 后端服务运行中且数据库已初始化，**When** 管理员发送 `POST /api/v1/auth/login` 携带正确邮箱密码，**Then** 返回 JWT Token 和用户信息，后续请求携带 Token 可访问受保护接口。
2. **Given** 已认证用户，**When** 发送 `POST /api/v1/requirements` 创建新需求，**Then** 需求存入数据库并自动生成编号（如 LMT-2026-001），返回完整需求信息。
3. **Given** 数据库有 50 条需求，**When** 请求 `GET /api/v1/requirements?page=1&pageSize=10&status=进行中`，**Then** 返回第 1 页共 10 条进行中的需求，以及总数和分页信息。
4. **Given** 存在一条 LMT 类型需求，**When** 调用转换接口将其转为 IR，**Then** 创建新 IR 需求并关联原 LMT，LMT 状态更新。
5. **Given** 存在一条 IR 需求，**When** 调用拆解接口提交 3 个子需求项，**Then** 批量创建 3 条 SR 需求且 parentId 指向该 IR。
6. **Given** 存在一条已完成的任务，**When** 尝试将其状态改为"进行中"，**Then** 返回 400 错误提示"已完成的任务不可变更状态"。

### User Story 2 - 前端 API 对接 (Priority: P2)

前端所有页面不再从 `lib/mock-data.ts` 读取数据，改为调用后端 REST API。用户在页面上的操作（创建、编辑、删除、筛选、分页、导出）实时与后端交互，操作结果立即反映在界面上。登录从硬编码验证改为 JWT Token 认证流程。

**Why this priority**: 前端对接 API 后才能将 P1 的后端能力呈现给用户。虽然 UI 样式还是旧的，但数据已经是真实的，用户可开始实际使用系统管理数据。

**Independent Test**: 启动前端和后端服务，在浏览器中登录并操作各个页面（需求管理、任务管理、项目管理），确认所有数据来自后端 API，创建/编辑/删除操作后数据正确更新。

**Acceptance Scenarios**:

1. **Given** 用户在登录页输入正确邮箱密码，**When** 点击登录，**Then** 获得 Token 并跳转到工作台，后续所有请求自动携带 Token。
2. **Given** 用户在需求管理页，**When** 选择类型筛选为"IR"并切换页码，**Then** 表格显示后端返回的 IR 类型需求分页数据，页面切换时重新请求而非客户端过滤。
3. **Given** 用户在需求列表页点击删除一条需求，**When** 确认删除，**Then** 调用后端删除接口，成功后刷新列表，该需求不再出现。
4. **Given** 用户 Token 过期或无效，**When** 发起任何 API 请求，**Then** 收到 401 响应后自动清除 Token 并跳转到登录页。
5. **Given** 用户在需求详情页查看一条 IR 需求，**When** 页面加载，**Then** 显示该 IR 的完整信息及其下级 SR/AR 需求树。

### User Story 3 - 前端 UI 企业级重设计 (Priority: P3)

系统整体界面升级为企业级 SaaS CRM 风格。包括全新设计的顶部导航栏、可折叠侧边栏、KPI 统计卡片、看板视图、图表、更精致的表格和表单样式。所有页面具备统一的视觉语言（配色、字体、间距、圆角、阴影）。

**Why this priority**: UI 重设计提升用户体验和工作效率，但不影响数据功能。P1+P2 完成后系统已可用，P3 在此基础上强化视觉表现和交互体验。

**Independent Test**: 在浏览器中浏览各页面，逐一验证新版布局组件（导航栏、侧边栏、KPI 卡片、看板视图、图表）渲染正确，动画流畅，响应式断点适配正常。

**Acceptance Scenarios**:

1. **Given** 用户登录后进入任意主页面，**When** 页面加载完成，**Then** 看到顶部固定导航栏（含 Logo、主导航链接、搜索框、通知图标、用户头像）和左侧可折叠侧边栏。
2. **Given** 用户在工作台页面，**When** 页面加载，**Then** 看到 4 个 KPI 统计卡片（需求总数、进行中数量、完成率、逾期任务数）和图表区域（需求趋势折线图、类型分布环形图）。
3. **Given** 用户在需求管理页，**When** 切换到看板视图，**Then** 需求按状态分列显示为可拖拽卡片，卡片含编号、名称、优先级标签和截止日期。
4. **Given** 用户在项目管理页，**When** 页面加载，**Then** 项目以卡片网格形式展示，每张卡片显示项目名称、财务编号、版本/迭代/成员/需求数量统计和状态标签。
5. **Given** 用户在手机浏览器打开系统，**When** 页面宽度小于 768px，**Then** 导航栏变为汉堡菜单，侧边栏隐藏，KPI 卡片全宽堆叠，表格转为卡片列表。

### Edge Cases

- 网络中断时 API 请求失败，前端应显示错误状态及重试按钮，而非白屏或无限加载。
- 分页数据在用户删除当前页最后一条记录时，应自动回退到上一页（首页除外）。
- 并发操作（如两人同时编辑同一条需求）时，后提交者应通过 updated_at 时间戳检测冲突，返回 409 错误提示"数据已被其他用户修改，请刷新后重试"，而非静默覆盖。
- CSV 导出数据量较大（超过 10000 条）时，后端应支持流式输出，前端应有下载进度反馈。
- 数据库连接断开时，后端应返回 500 错误并记录日志，而非崩溃。
- 用户直接 URL 访问受保护页面（未登录状态），应被重定向到登录页。
- 需求编号生成在并发创建时不应产生重复编号。

## Requirements *(mandatory)*

### Functional Requirements

**后端 API 系统**

- **FR-001**: 系统 MUST 提供基于 JWT 的用户认证接口（登录、获取当前用户信息），所有业务接口 MUST 验证 Token 有效性。
- **FR-002**: 系统 MUST 为需求管理提供完整 REST API：列表查询（支持类型/状态/关键词筛选[搜索字段: name+code+description]和分页）、详情查看、创建、编辑、逻辑删除。
- **FR-003**: 系统 MUST 支持 LMT 转 IR（创建子需求并关联）、需求拆解（IR→SR 或 SR→AR 批量创建子需求）、IR 需求树（递归嵌套结构）、子需求列表和链路追溯。
- **FR-004**: 系统 MUST 为任务管理提供完整 REST API：列表查询（支持类型/状态/关键词[搜索字段: name+code+description]/分配人筛选、排序和分页）、统计概览、CRUD 操作。
- **FR-005**: 系统 MUST 支持任务分配（指定负责人并自动变更状态）和任务状态变更，状态变更 MUST 遵循状态机规则（已完成/已关闭不可回退）。
- **FR-006**: 系统 MUST 为项目管理提供完整 REST API：列表（含版本/迭代/成员/需求数量统计）、详情（含统计+成员+日志）、CRUD、成员管理、项目结构树。
- **FR-007**: 系统 MUST 为版本管理和迭代管理提供完整 REST API：列表（含统计字段）、详情（含关联数据+日志）、CRUD、CSV 导出。
- **FR-008**: 系统 MUST 为测试用例提供 REST API：列表、统计（通过率/执行率）、CRUD、单条和批量状态更新。
- **FR-009**: 系统 MUST 为工作台提供数据接口：紧急需求列表（按优先级和截止日期排序）和逾期任务列表。
- **FR-010**: 系统 MUST 使用统一响应格式 `{ code, message, data }`，分页接口统一使用 `{ page, pageSize, total, list }` 结构。
- **FR-011**: 系统 MUST 对所有创建/更新接口进行参数校验，校验失败返回 400 及具体错误描述。
- **FR-012**: 系统 MUST 在创建需求、任务、测试用例时自动生成唯一编号（格式分别为 `{TYPE}-{YEAR}-{NNN}`、`TASK-{YEAR}-{NNN}`、`TC-{YEAR}-{NNN}`）。
- **FR-027**: 系统 MUST 通过 Filter 或 AOP 记录所有 API 请求的 HTTP 方法、路径、状态码和处理耗时，日志输出到控制台。

**前端 API 对接**

- **FR-013**: 前端 MUST 创建统一 HTTP 客户端模块，自动附加 JWT Token、处理 401 跳转登录、统一错误处理。
- **FR-014**: 前端 MUST 将所有数据获取从 `lib/mock-data.ts` 同步调用替换为异步 API 调用，每个数据获取页面 MUST 包含 loading、error、empty 三种状态处理。
- **FR-015**: 前端 MUST 将客户端分页/筛选/排序逻辑改为服务端查询参数传递。
- **FR-016**: 前端 MUST 在 Token 缺失或过期时自动重定向到登录页，在受保护路由中检查认证状态。
- **FR-017**: 前端 MUST 支持 CSV 导出功能，通过调用后端导出接口下载文件。

**前端 UI 重设计**

- **FR-018**: 系统 MUST 采用统一设计系统：主色调蓝色、8px 间距体系、统一圆角和阴影规范。
- **FR-019**: 系统 MUST 重构布局外壳：顶部固定导航栏（含 Logo、主导航、搜索框/⌘K命令面板[搜索页面路由 + 需求/任务/项目名称和编号]、通知图标[仅UI占位，点击显示"暂无通知"空状态，无后端API]、用户菜单）、可折叠侧边栏（窄/宽两种状态）、底部页脚。
- **FR-020**: 工作台 MUST 包含 KPI 统计卡片区、需求趋势图表区、紧急需求列表和动态时间线。
- **FR-021**: 需求管理页 MUST 支持表格视图和看板视图切换，看板视图按状态分列。
- **FR-022**: 需求详情页 MUST 支持面包屑导航、需求信息卡、IR/SR/AR 递归折叠树、测试用例统计和进度条。
- **FR-023**: 任务管理页 MUST 支持看板视图（按状态分列，支持拖拽变更状态）、表格视图、统计概览卡片。
- **FR-024**: 项目管理页 MUST 以卡片网格展示项目列表，项目详情页 MUST 包含概览/版本/迭代/成员/日志标签页切换。
- **FR-025**: 系统 MUST 支持响应式布局：大屏完整布局、中屏侧边栏折叠、小屏隐藏侧边栏、手机端汉堡菜单+全宽卡片。
- **FR-026**: 系统 MUST 实现页面过渡动画、卡片悬浮效果、侧边栏展开动画、看板拖拽效果。

### Key Entities

- **需求 (Requirement)**: 核心实体，包含编号、名称、类型（LMT/IR/SR/AR）、客户、项目、期望日期、状态、优先级、描述。通过 parentId 形成层级分解关系，IR 通过 irId 关联回 LMT，AR 通过 iterationId 归属迭代。
- **任务 (Task)**: 包含编号、名称、类型、分配人、创建人、截止日期、状态、关联需求。拥有操作历史记录。
- **项目 (Project)**: 包含名称、编号、财务编号、负责人、起止日期、状态。关联多个版本、迭代、成员。
- **版本 (Version)**: 归属项目，包含产品名称、版本号、起止日期、状态。
- **迭代 (Iteration)**: 归属项目和版本，包含名称、起止日期、状态。
- **测试用例 (TestCase)**: 归属 AR 需求，包含编号、名称、状态（通过/失败/未执行）、优先级、执行日期。
- **用户 (User)**: 包含用户名、邮箱、加密密码、真实姓名，用于认证和操作记录。
- **操作日志 (OperationLog)**: 记录项目/版本/迭代的操作历史，包含目标类型、目标 ID、操作类型、操作人、变更前后值。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户登录后可在 3 秒内加载完成任意列表页（含 10 条数据），包含完整的导航栏和侧边栏渲染。
- **SC-002**: 所有 CRUD 操作（创建/编辑/删除）在 2 秒内完成并反馈结果给用户（成功提示或错误信息）。
- **SC-003**: 需求看板视图支持同时展示 100 张卡片而不出现明显卡顿。
- **SC-004**: 系统在 3 种主流屏幕尺寸（1920px / 1366px / 375px）下布局正确，无横向滚动条，核心功能可正常操作。
- **SC-005**: API 接口在 100 并发请求下，95% 的请求在 1 秒内返回响应。
- **SC-006**: 后端支持至少 10,000 条需求数据量下，列表查询（含筛选和分页）在 500ms 内返回。
- **SC-007**: 用户可在不查阅帮助文档的情况下，完成创建需求 → 转换为 IR → 拆解为 SR → 拆解为 AR 的完整层级分解流程。

## Assumptions

- 目标用户为研发团队内部成员（产品经理、项目经理、开发工程师、测试工程师），具备基本软件操作能力。
- 用户使用的主流浏览器为 Chrome、Edge、Firefox 最新两个版本，不支持 IE 浏览器。
- 数据库 MySQL 服务器（mysql6.sqlpub.com:3311）在开发期间可用且网络可达，字符集为 utf8mb4，时区为 Asia/Shanghai。
- 后端服务部署在本地或内网环境，前端通过 localhost:3000 访问，后端通过 localhost:8080 访问。
- 默认管理员账号为 admin@example.com，初始密码在数据库初始化脚本中设置。
- 系统仅支持单一团队使用，暂不考虑多租户、组织架构和权限分级。
- 本次重构不涉及移动端原生 App，仅需支持移动端浏览器响应式访问。
- 所有删除操作均为逻辑删除（标记删除而非物理删除），数据可恢复。
- 旧版 mock-data.ts 文件在对接 API 后将保留不删，以备参考。
